# Review, Approve, and Export Generated Work

This document describes the review, approval, rejection, and export workflow
for artifacts generated by AIP. This is a spec-required vertical slice that
composes existing AIP primitives — no parallel systems are created.

## ECS Lifecycle States Used

The existing AIP artifact lifecycle states are:

| State | Description |
|-------|-------------|
| SPECIFIED | Initial requirements state |
| GENERATED | Artifact created by `aip ask --save-artifact`, pending review |
| REVIEWED | Quality gate passed (intermediate state) |
| APPROVED | DEFINER-approved, written to canonical store |
| REJECTED | DEFINER rejected, artifact preserved for audit |
| SUPERSEDED | Replaced by newer version (terminal) |
| FAILED | Generation or evaluation failed |

**Transition path for approval:**
```
GENERATED → REVIEWED → APPROVED
```

When a DEFINER approves an artifact via `aip review approve`, AIP performs
two ECS transitions: first from GENERATED to REVIEWED (quality gate pass),
then from REVIEWED to APPROVED (canonical promotion). The artifact is also
written to the CanonicalStore with `approved_by="definer"`.

**Transition path for rejection:**
```
GENERATED → REJECTED
```
or
```
REVIEWED → REJECTED
```

Rejected artifacts are preserved (never deleted) and can be re-generated
later via the REJECTED → GENERATED re-synthesis loop.

**Needs-revision verdict:**
NEEDS_REVISION is a ReviewVerdict value, not an ECS state. When a DEFINER
marks an artifact as needing revision, the artifact stays in its current
ECS state (typically GENERATED) and the revision instruction is stored as
a review event for later reference.

## Complete Example Flow

### Step 1: Ingest conversations

```bash
aip ingest directory ./my_threads --project aip_loom
```

### Step 2: Ask a question and save the answer as an artifact

```bash
aip ask "What have we decided about AIP_Loom artifact storage?" \
  --project aip_loom \
  --source all \
  --show-context \
  --save-artifact
```

This produces an artifact in GENERATED state (pending DEFINER review).
The output includes the artifact ID, which you need for the next steps.

### Step 3: List artifacts pending review

```bash
aip review list --project aip_loom
```

Shows all GENERATED artifacts for the project with:
- Artifact ID
- Title (originating prompt)
- Lifecycle state
- Created timestamp
- Source count
- Session ID
- Model slot/provider

### Step 4: Inspect the artifact content

```bash
aip review show <artifact_id>
```

Displays:
- Full artifact content
- Lifecycle state
- Originating prompt
- Model slot/provider
- Source count
- Review notes/history
- Export eligibility

### Step 5: Inspect source grounding

```bash
aip review sources <artifact_id>
```

Shows the provenance/source links created by `aip ask --save-artifact`:
- Source ID or chunk ID
- Source type (conversation_chunk, artifact, etc.)
- Source title/file/conversation name
- Content snippet
- Metadata for tracing back to the original ingested content

This proves the generated answer was grounded in the right prior
conversations and artifacts.

### Step 6: Approve the artifact

```bash
aip review approve <artifact_id>
```

Transitions the artifact through the existing ECS lifecycle:
GENERATED → REVIEWED → APPROVED. Writes to CanonicalStore with DEFINER
sovereignty. Records the approval as a review event.

### Step 7: Export to markdown

```bash
aip export artifact <artifact_id> \
  --format markdown \
  --out ./exports/aip_loom_artifact_storage.md
```

Exports the artifact content to a markdown file with:
- YAML frontmatter (artifact ID, project, lifecycle state, timestamps,
  model info, source count)
- Full artifact content
- Provenance footer listing all source links

## Review Commands Reference

### `aip review list --project <name>`

List artifacts pending review for a project. By default shows GENERATED
artifacts. Use `--state` to filter by other states.

Options:
- `--project` (required): Project name
- `--state`: Filter by lifecycle state (can be specified multiple times; default: GENERATED)
- `--db-path`: SQLite database path (default: `db/state.db`)

### `aip review show <artifact_id>`

Show artifact details including content, lifecycle state, and review history.

### `aip review sources <artifact_id>`

Display source/provenance links for an artifact. Shows where the generated
answer's content came from.

### `aip review approve <artifact_id>`

Approve a generated artifact. Transitions: GENERATED → REVIEWED → APPROVED.
Writes to CanonicalStore. No auto-approve — this IS the DEFINER gate.

Validation:
- Artifact must exist
- Content must not be empty
- Source links must exist for `ask_answer` type artifacts

### `aip review reject <artifact_id> --note "<reason>"`

Reject a generated artifact. The artifact is preserved (never deleted).
Source links are preserved. The reviewer note is stored.

Options:
- `--note`: Reason for rejection

### `aip review needs-revision <artifact_id> --note "<instruction>"`

Mark artifact as needing revision. The artifact stays in its current ECS
state. The revision instruction is stored as a review event.

Options:
- `--note`: Revision instruction for the artifact

## Export Commands Reference

### `aip export artifact <artifact_id> --format markdown --out <path>`

Export an artifact to a markdown file.

Behavior:
- Refuses to export REJECTED artifacts by default
- Refuses to export unapproved (GENERATED/REVIEWED) artifacts without `--force`
- Creates parent directories if needed
- Never silently writes an empty file
- Includes metadata frontmatter and provenance footer

Options:
- `--format`: Export format (currently only "markdown")
- `--out` (required): Output file path
- `--force`: Force export of rejected or unreviewed artifacts
- `--db-path`: SQLite database path (default: `db/state.db`)

### `aip export project <project_name> --format markdown --out <path>`

Export all approved artifacts for a project as a single markdown bundle.

Behavior:
- Includes APPROVED artifacts by default
- Excludes REJECTED artifacts always
- Excludes unreviewed (GENERATED/REVIEWED) artifacts unless `--include-unreviewed`
- Produces a markdown file with an artifact index section
- Includes provenance metadata compactly

Options:
- `--format`: Export format (currently only "markdown")
- `--out` (required): Output file path
- `--include-unreviewed`: Include GENERATED/REVIEWED artifacts
- `--db-path`: SQLite database path (default: `db/state.db`)

## How Review Notes Are Stored

Review notes are stored as events in the EventStore with `event_type="review_verdict"`.
Each event includes:
- `verdict`: APPROVED, REJECTED, or NEEDS_REVISION
- `detail`: The reviewer note or instruction
- `actor`: "definer" (the human operator)
- `from_state` and `to_state`: ECS state transitions
- Additional metadata (rejection_note, revision_instruction)

## How Source/Provenance Links Are Inspected

Source links are stored in the artifact metadata (written by `aip ask --save-artifact`):
- `source_ids`: List of chunk IDs or artifact IDs used as sources
- `source_types`: Corresponding source types (conversation_chunk, artifact, etc.)

The `aip review sources` command reads this metadata and attempts to look up
each source in the ArtifactStore for additional details (title, format, domain).

## Which Artifacts Are Exportable by Default

| Lifecycle State | Exportable | Notes |
|----------------|------------|-------|
| APPROVED | Yes | Default export target |
| REVIEWED | With `--force` | Quality gate passed but not yet canonical |
| GENERATED | With `--force` | Pending review |
| REJECTED | With `--force` | Requires explicit override |
| FAILED | No | Generation failed, no content |
| SUPERSEDED | With `--force` | Replaced by newer version |

## How to Export Draft/Generated Work

Use the `--force` flag:

```bash
aip export artifact <artifact_id> \
  --format markdown \
  --out ./draft_export.md \
  --force
```

## How Project Export Is Structured

Project export produces a single markdown file with:

1. **Header**: Project name, export timestamp, total artifact count
2. **Artifact Index**: Numbered list of all exported artifacts with ID, title, and state
3. **Artifact Sections**: Each artifact in full, with its own frontmatter and provenance footer

Example structure:
```markdown
# Project: aip_loom

Exported: 2025-01-15T10:30:00+00:00
Total artifacts: 3

## Artifact Index

1. `ask:abc123` — What have we decided about artifact storage? [APPROVED]
2. `ask:def456` — Architecture decisions for the storage layer [APPROVED]
3. `ask:ghi789` — API design for the canonical store [APPROVED]

---

## Artifact: ask:abc123
[full artifact content with frontmatter and provenance]

---
[...more artifacts...]
```
