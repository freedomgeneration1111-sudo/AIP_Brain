# Ask AIP Using Ingested Conversations and Project Memory

The `aip ask` command provides a source-grounded question-answering work loop
that uses ingested conversations and existing project artifacts as context.
Every generated answer is grounded in retrieved sources with full provenance.

## How This Builds on `aip ingest`

The ask pipeline reuses the same persistent stores that `aip ingest` writes to:

| Store | Implementation | Persistence | Used by ingest | Used by ask |
|-------|---------------|-------------|----------------|-------------|
| ArtifactStore | VersionedArtifactStore | SQLite (data/aip.db) | Writes conversation artifacts | Reads artifacts, saves answers |
| LexicalStore | SqliteFts5LexicalStore | SQLite FTS5 (data/lexical.db) | Indexes conversation chunks | Primary search backend |
| VectorStore | InMemoryVectorStore | Ephemeral | Indexes chunks (when embedding available) | Supplementary semantic search |
| EventStore | QueryableEventStore | SQLite (data/aip.db) | Records ingestion events | Records ask traces |
| ProjectStore | SqliteProjectStore | SQLite (data/aip.db) | N/A | Resolves projects |
| EcsStore | PersistentEcsStore | SQLite (data/aip.db) | N/A | Lifecycle transitions for saved artifacts |

**Critical design choice**: LexicalStore (FTS5) is the primary search backend
because it is persistent. Content ingested via `aip ingest` survives process
restarts in the FTS5 database. VectorStore is supplementary because the
default implementation is in-memory and ephemeral.

## Usage

### Basic ask

    aip ask "What is the current state of CodeForge?" --project codeforge

### Search all sources and save as artifact

    aip ask "What have we decided about AIP_Loom artifact storage?" \
      --project aip_loom \
      --source all \
      --save-artifact

### Show context before generation

    aip ask "Draft the next architecture section from prior discussions" \
      --project aip_loom \
      --show-context \
      --save-artifact

### Limit sources and use specific model slot

    aip ask "Summarize the API design decisions" \
      --project api_project \
      --max-sources 5 \
      --model-slot synthesis

## Options

| Option | Default | Description |
|--------|---------|-------------|
| `--project` | (required) | Project name to search within |
| `--source` | `all` | Source filter: `ingested`, `artifacts`, or `all` |
| `--max-sources` | `10` | Maximum number of sources to retrieve |
| `--save-artifact` | `false` | Save the answer as a draft artifact |
| `--model-slot` | `synthesis` | Model slot to use for generation |
| `--show-context` | `false` | Display retrieved context before generation |
| `--session` | (auto-generated) | Session ID for continuity |
| `--db-path` | `data/aip.db` | SQLite database path |

## Source Grounding

Every generated answer maintains provenance back to the retrieved sources.

Source references include:
- Source ID (chunk ID or artifact ID)
- Source type (`conversation_chunk`, `project_artifact`, `compiled_knowledge`)
- Title (conversation title or artifact name)
- Retrieval score
- Content snippet
- Domain

Answers include inline source citations in the format:

    [source: <conversation_id>/<chunk_id>]

or for non-conversation sources:

    [source: <source_id>]

### Source Filtering

- `--source ingested`: Only search ingested conversation chunks (type=`conversation_chunk`)
- `--source artifacts`: Only search project artifacts and non-conversation content
- `--source all`: Search everything (default)

## How Generated Artifacts Are Stored

When `--save-artifact` is provided:

1. The answer is saved to ArtifactStore with metadata including:
   - `artifact_type`: `"ask_answer"`
   - `project_id` and `project_name`
   - `prompt`: the original question
   - `source_ids`: list of all source chunk/artifact IDs used
   - `source_types`: corresponding source type for each
   - `session_id`: the session reference
   - `model_slot` and `model_name`: model used for generation

2. An ECS transition is recorded: `None → GENERATED`
   - Actor: `ask_pipeline`
   - Reason: `"Generated by ask pipeline — pending DEFINER review"`
   - The artifact is NOT auto-approved
   - It remains in `GENERATED` state until reviewed

3. The saved artifact is also indexed in LexicalStore so future
   ask queries can find it via `--source artifacts`

## How to Review Generated Artifacts

After `--save-artifact`, the artifact is in `GENERATED` state (draft, pending review).

To review:
1. Find the artifact ID from the ask output
2. Inspect the artifact content and sources
3. Use the existing review/lifecycle mechanisms to approve or reject

The artifact will not be promoted to canonical without explicit review approval,
following the existing ECS lifecycle: SPECIFIED → GENERATED → REVIEWED → APPROVED.

## What Happens When No Model Provider Is Configured

If no model provider is configured (no `AIP_SYNTHESIS_BASE_URL` or config entry):

- AIP returns `NEEDS_CONFIGURATION` status
- No fake answer is generated
- Retrieved sources are still displayed so the user can inspect what would have been used
- The user can configure the model provider by:
  - Setting `AIP_SYNTHESIS_BASE_URL` and `AIP_SYNTHESIS_MODEL` environment variables
  - Adding a `[models.synthesis]` section to `config/aip.config.toml`

## Session Trace

Every ask query (successful or failed) records a session trace in EventStore:

- Event type: `ask_query`
- Session ID: unique per ask invocation (or provided via `--session`)
- Project ID: the resolved project
- Prompt: the question asked
- Source count and source IDs
- Model slot and provider used
- Answer digest (SHA-256 hash)
- Whether an artifact was saved
- Any errors encountered

This enables later inspection of:
- What was asked
- What context was used
- What answer was generated
- What artifact was saved
- What sources grounded it

## Failure Modes

| Condition | Status | Behavior |
|-----------|--------|----------|
| Project not found | `NO_PROJECT` | Clear error, no orphan artifacts |
| No ingested/project material | `NO_PROJECT_MEMORY` | Clear message, suggest `aip ingest` |
| No relevant sources found | `NO_PROJECT_MEMORY` | Clear message, no fabricated sources |
| No model configured | `NEEDS_CONFIGURATION` | Sources shown, no fake answer |
| Model call fails | `MODEL_FAILURE` | Sources preserved, no artifact saved, failure traced |
| Artifact save fails | `ARTIFACT_SAVE_FAILURE` | Answer still displayed, failure traced |

## Architecture

The ask pipeline follows the three-layer AIP architecture:

```
CLI (click)                         Tests (pytest)
  │                                    │
  aip ask ────┐                       ├─ TestAskWithMarkdownContent
              │                       ├─ TestAskWithChatGPTContent
              ▼                       ├─ TestSourceFilter*
     create_ask_stores()             ├─ TestNoModelConfigured
              │                       ├─ TestSaveArtifact
              ▼                       ├─ TestArtifactProvenance
     ask()                          ├─ TestSessionTrace
    ┌─────────┼──────────┐          ├─ TestModelFailure
    ▼         ▼          ▼          ├─ TestArtifactSaveFailure
 Resolve   Search     Dispatch      ├─ TestSourceReferencesPersistence
 Project   Sources    to Model      └─ TestAskWithRealLexicalStore
    │         │          │
    ▼         ▼          ▼
 ProjectStore  LexicalStore  ModelProvider
               VectorStore   (ModelSlotResolver)
               ArtifactStore
               EventStore
               EcsStore
```

No parallel storage system — ask reuses the exact same stores that ingest writes to.
